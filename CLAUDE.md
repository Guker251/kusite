

## технические SEO-проверки

проводить при каждом аудите сайта (не при обработке отдельной страницы):

### robots.txt
- проверить наличие `Disallow` для сервисных директорий: /articles/faq/, /articles/karta-sayta/, /articles/knigi/, /articles/stati/;
- убедиться, что `Sitemap:` указывает на актуальный файл;
- не закрывать контентные разделы;

### sitemap.xml
- не включать сервисные страницы (КУ-методология, тесты, служебные);
- не включать 301-редиректы — только канонические URL;
- все хабы и контентные статьи должны присутствовать;
- URL раздела признаков алкоголизма: `/articles/priznanki-alkogolizma/` (не uznaki);
- обновлять `<lastmod>` при изменении страницы;

### noindex на сервисных страницах
- все страницы с типом `service` в реестре обязаны иметь `<meta name="robots" content="noindex, nofollow">`;
- проверка: `grep -L "noindex" articles/*.html` должна возвращать только контентные файлы;

### canonical на контентных страницах
- каждая контентная страница обязана иметь `<link rel="canonical" href="...">`;
- canonical должен совпадать с og:url и URL в BreadcrumbList JSON-LD;
- для директорий: canonical всегда с trailing slash;

### 301-редиректы
- редиректы прописаны в `.htaccess`;
- при переименовании раздела добавлять новый Redirect 301;
- использовать полный URL назначения (включая https://alkonorm.ru);
- проверять, что целевой URL существует и не возвращает 404;

команда:
```bash
curl -s -X POST \
  "https://api.webmaster.yandex.net/v4/user/752011635/hosts/https:alkonorm.ru:443/recrawl/queue" \
  -H "Authorization: OAuth y0__xDzksvmAhjMzj4g7fW_1BbMIeHaAwV9hj5l0yHGBEECRTQ9lw" \
  -H "Content-Type: application/json" \
  -d "{\"url\": \"<URL страницы>\"}"
```

условия:
- выполнять только после успешного деплоя;
- при ошибке API не останавливать обработку следующих страниц;
- не проверять квоту перед каждым запросом;

## использование semantic-core.yaml

в проекте используется полный файл семантического ядра:
- docs/semantic-core.yaml;

правила работы:

- не читать файл целиком;
- находить в semantic-core.yaml только блок, соответствующий текущему URL страницы;
- извлекать только:
  - основной запрос;
  - дополнительные запросы, относящиеся к этой странице;

- не использовать запросы, не относящиеся к текущей странице;

- если структура файла позволяет:
  - искать по URL;
  - если нет — по совпадению основного запроса;

- после извлечения:
  - использовать основной запрос как главный критерий проверки;
  - дополнительные запросы использовать для:
    - проверки полноты;
    - структуры;
    - подзаголовков;

- не держать в контексте лишние части semantic-core.yaml;

- если не удалось однозначно найти блок:
  - использовать только основной запрос из pages-status.md;

приоритет:

- данные из semantic-core.yaml важнее любых предположений;

## извлечение контента страницы

при анализе страницы:
- использовать только основной контент статьи;
- игнорировать:
  - меню;
  - футер;
  - сайдбары;
  - служебные блоки;
  - повторяющиеся элементы;

если доступен HTML:
- извлекать только текст статьи;
- не учитывать разметку, не влияющую на смысл;

## использование шаблона статьи

если в проекте есть файл article-skeleton.html:

- использовать его как эталон структуры статьи;
- проверять наличие ключевых блоков:
  - введение;
  - основные разделы;
  - списки при необходимости;
  - логичное завершение;

- не копировать шаблон дословно;
- использовать только как ориентир структуры;

## критерии приоритетов

считать проблемой уровня критично:

- страница не соответствует основному запросу;
- основной запрос отсутствует в title или H1;
- в первых абзацах нет ответа на запрос;
- основной интент не закрыт;
- структура мешает пониманию смысла;

считать проблемой уровня важно:

- слабая структура;
- недостаточная полнота;
- слабый первый экран;
- недостаточная перелинковка;
- избыточные или размытые фрагменты;

считать проблемой уровня желательно:

- улучшения читаемости;
- стилистические улучшения;
- дополнительные усиления структуры;

## перелинковка

при проверке:

- добавлять внутренние ссылки, если их нет;
- использовать только релевантные статьи;
- обеспечивать логичный переход пользователя;
- избегать случайных или формальных ссылок;

## тип страницы

учитывать поле type:

- article:
  - максимальный акцент на полноту ответа и структуру;

- hub:
  - акцент на навигацию и перелинковку;

- service:
  - акцент на понятность и действие пользователя;
